Core Thesis Traditional AI-assisted coding is reactive — you prompt, it responds. This system inverts that model: the AI team runs continuously on a cron schedule, reads its own state, picks the next task, delegates to specialists, and produces small focused pull requests. Humans steer by dropping directives into an inbox file. The AI does the rest.
🧠

Persistent Memory

Agents maintain state across runs via ai-state.json and MEMORY.md — structured JSON for machine state, markdown for knowledge accumulation. No context window wasted on re-discovery.

🔄

Continuous Progress

Every hour, the coordinator wakes up, reads the current state, and makes concrete forward progress — designing schemas, writing code, running tests, or reviewing PRs. No human trigger needed.

🛡️

Safe by Design

Agents create PRs but never merge them. Each PR is labeled, size-limited, and requires human approval. The system is autonomous but never unsupervised.

Orchestrator

🎯 Coordinator

Owns the sprint backlog. Reads state, checks the human inbox, decomposes work into subtasks, and delegates. Never writes code — only plans and tracks. Enforces the task lifecycle: proposed → ready → in_progress → review → done.

Specialist

📐 Architect

Designs system architecture, API contracts, database schemas, and module boundaries. Outputs structured specs (OpenAPI, SQL, JSON Schema) — never application code. Every decision includes a one-line rationale.

Specialist

💻 Coder

Implements features following architect specs. Edits only project/src/ and project/migrations/. Reads surrounding code before writing. One feature per task — no scope creep.

Specialist

🧪 Tester

Writes unit, integration, and acceptance tests. Reports results as structured JSON. Never modifies application code — reports failures back to the coordinator for delegation.

Specialist

🔍 Reviewer

Reviews for correctness, security (OWASP Top 10), and standards compliance. Returns structured findings with severity levels. Does not modify files unless explicitly authorized.

Key Principle

🚧 Strict Boundaries

Each agent has a defined filesystem scope and responsibility boundary. The coder can't modify tests. The reviewer can't change code. The tester can't alter the backlog. This prevents cascading errors and makes agent behavior predictable.

📊 Structured Output

Every heartbeat ends with a JSON status summary — tasks completed, in-progress, blocked, and PRs created. Machine-readable for dashboarding and monitoring.

🪙 Token Efficiency

Agents reference files with #file:path instead of pasting content. Structured JSON over prose. Minimal agent invocations per cycle. Each heartbeat costs only what it needs.

🔒 Safe Outputs

The workflow can only create PRs — never push directly to main, never merge, never delete. Every change goes through the PR gate with the ai-generated label for easy filtering.

Self-Healing Mechanisms

The most interesting pattern in this system: autonomous error detection and recovery without human intervention. When something breaks, the system diagnoses and fixes itself.

Design Principle A self-healing system doesn't just detect errors — it has pre-defined recovery paths for common failure modes. The key insight is that most failures in autonomous AI systems are predictable and recoverable: state corruption, task conflicts, stale context, and drift from specifications.

Recovery Mechanisms

🔄 State File Recovery

If ai-state.json becomes malformed or inconsistent (e.g., a task stuck in in_progress for multiple heartbeats), the coordinator detects the anomaly, resets the task to ready, and logs the recovery in decisions_log. No manual intervention needed.

Trigger: Stale task age > 3 heartbeats

🧪 Test-Driven Rollback

When the tester agent reports failures after a coder implementation, the coordinator doesn't just flag it — it routes the failure details back to the coder with specific instructions to fix. If the fix fails twice, the task is reverted and re-queued with a redesign flag for the architect.

Trigger: Test failure after implementation

📋 Spec-Code Reconciliation

The reviewer agent detects when implementation drifts from architect specifications. Rather than just flagging it, the system generates a reconciliation task that compares the spec (OpenAPI, schema) against the actual code and produces a targeted fix list.

Trigger: Review finding type = "spec_drift"

🧹 Memory Compaction

As MEMORY.md grows, older entries become noise. The coordinator periodically summarizes old sprint entries into a compact "historical decisions" section, preserving key learnings while keeping the active context window focused on current work.

Trigger: MEMORY.md > token threshold

🔀 Conflict Resolution

When two agent outputs conflict (e.g., architect changes a schema that the coder just implemented against), the coordinator mediates: it compares timestamps, checks which output aligns with the latest human directive, and routes the losing side for rework.

Trigger: Overlapping file modifications

🚨 Deadlock Detection

If no task progresses for multiple consecutive heartbeats, the coordinator recognizes the deadlock, diagnoses the root cause (circular dependency, missing prerequisite, unclear acceptance criteria), and either redecomposes the blocked task or escalates via the human inbox response channel.

Trigger: Zero progress for 3+ heartbeats
Why This Matters In traditional workflows, any agent error halts the pipeline until a human investigates. Self-healing means the system can recover from the majority of failures autonomously — reducing the human oversight burden from constant monitoring to periodic review. The agents operate like an on-call team that fixes its own incidents.

Priority System

🎚️ Four Priority Levels

  • Critical — Drop everything. Current task paused immediately.
  • High — Processed before any existing backlog items.
  • Normal — Folded into the current sprint at natural priority.
  • Low — Noted for future sprints; no immediate action.
Message Types

📨 Four Directive Types

  • Directive — Direct order: "Build X" or "Use technology Y".
  • Question — Team researches and answers in a response section.
  • Feedback — Commentary on recent work; coordinator adjusts approach.
  • Pivot — Major direction change. Halts current work, re-plans sprint.

Agentic Programming Patterns

Key Concepts Explored

This project is a working laboratory for modern agentic programming patterns. Each concept below is implemented and tested in the system.

Concept What It Means How It's Implemented Here
Persistent Agent Memory Agents retain knowledge across invocations without relying on conversation history MEMORY.md for knowledge, ai-state.json for structured state — filesystem as shared memory
Task Decomposition Breaking complex work into small, well-defined subtasks with clear acceptance criteria Coordinator decomposes epics into ≤1hr tasks with explicit criteria and file-scope constraints
Agent Specialization Each agent has a narrow, well-defined role with strict boundaries on what it can do 5 agents with filesystem-scoped permissions: coder edits src/, tester edits tests/, etc.
Self-Healing Recovery The system detects its own errors and has pre-defined recovery paths for common failures Stale task detection, test-driven rollback, spec-code reconciliation, deadlock breaking
Structured Handoffs Agents communicate through well-defined JSON contracts, not free-form text Every agent outputs typed JSON (task_id, files_modified, verdict, etc.) for deterministic routing
Safe Outputs / Human Gating AI autonomy bounded by safety constraints — PRs created but never merged autonomously safe-outputs in workflow config: labeled PRs, size limits, no direct pushes to main
Token Efficiency Minimizing LLM token consumption through references, structured output, and dynamic tool loading File references via #file:path, JSON responses, minimal per-cycle agent invocations
Asynchronous Human Control Humans steer without being in the loop for every decision — async communication via inbox INBOX.md with priority levels and directive types; processed on each heartbeat cycle

Project structure, base configuration, PostgreSQL schema design, JWT authentication module with refresh tokens, API foundation with middleware and error handling.

Sprint 2 — Core Domain Logic

Implement the core business domain entities, CRUD operations, validation rules, and the multi-tenant data isolation layer.

Sprint 3 — Integration & APIs

External API integrations, webhook handlers, rate limiting, structured logging (pino), and the health check endpoint.

React frontend with server components, dashboard views, responsive design, and end-to-end test suite with Playwright.
